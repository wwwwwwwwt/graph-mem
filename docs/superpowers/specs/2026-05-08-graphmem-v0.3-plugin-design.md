## 1. 现状诊断（诚实评估）

v0.2 后台 4-session 测试（28 turns）暴露的问题：

| 问题 | 数据 | 影响 |
|---|---|---|
| 写入阻塞 | 945s 写 28 turns | 不可用，每轮对话都要等 LLM |
| 重复压缩 | 4 session → 16 episode | 同一个 session 被反复压缩，episode 质量极差 |
| 召回准确率 | 中文 50% (3/6) | 注入 Claude prompt 的上下文大量漏掉关键信息 |
| Entity 冗余 | 82 个，无去重 | 同一个 PostgreSQL 出现 N 次，图膨胀 |
| key_points 乱码 | Unicode 转义 | 压缩结果不可读 |

**根本原因**: 写入和压缩耦合在同一个事务中，每次 `write_turn` 都同步调用 LLM。

---

## 2. 核心决策：写入与压缩彻底解耦

### 2.1 旧架构（v0.2，有问题）

```
write_turn() → 创建 L0 → [触发压缩] → 调用 LLM → 生成 L1 + L2
               ↑___________________________|
                      同步阻塞，同一线程
```

### 2.2 新架构（v0.3，已实现）

```
write_turn() → 创建 L0 + embed → 写入 compression_queue → 返回 (<50ms)
                                                    ↓
                              [Background Worker]  ← 混合触发（5min 或 10 条）
                                                    ↓
                                      按 session 分组读取 queue
                                                    ↓
                                      批量调用 LLM 一次
                                      → 生成 L1 episode + L2 entity
                                      → 创建边 (DERIVED_FROM, MENTIONS)
```

**关键机制**:
- **Session 压缩锁**: 一个 session 正在被压缩时，新写入的 turns 进 queue 等待。防止同一个 session 被多个 worker 同时处理。
- **批量压缩**: Worker 一次读取一个 session 的所有未压缩 turns，只调一次 LLM API。
- **队列持久化**: SQLite 队列保证进程重启不丢数据。

## 3. 召回架构：精准度 + 长期 + 效率

### 3.1 三层召回

| 层级 | 触发方式 | 数据源 | 延迟 | 用途 | 状态 |
|---|---|---|---|---|---|
| **L1 极速召回** | pre-prompt hook / MCP | 向量检索 L1 episode | <100ms | 自动注入 Claude prompt | Phase 3 |
| **L2 深度召回** | MCP tool_call / CLI | 向量 + 图扩散 2 跳 | ~1s | 用户主动查询 | ✅ Phase 1 |
| **L3 长期归纳** | 夜间异步 | 跨 session reflection | 后台 | 归纳技术偏好和模式 | Phase 3 |

### 3.2 精准度保证

**A. 多语言 Embedding** ✅
- 替换 `all-MiniLM-L6-v2` → `BAAI/bge-small-zh-v1.5`（中文效果更好）
- 已实现于 `BGEEmbedClient`，已接入 `Memory.open` 工厂

**B. Re-rank 层** (Phase 2)
- 向量检索先取 top-20，然后用轻量 cross-encoder 重排，取 top-5
- cross-encoder 可以用本地小模型（如 `BAAI/bge-reranker-base`），零 API 成本

**C. 时间衰减** (Phase 2)
- 召回打分加入时间因子: `score = vec_score * exp(-age_days / 30)`
- 30 天前的记忆权重衰减到 37%，避免旧项目干扰新项目

**D. 图扩散** ✅
- L2 深度召回时，从向量命中的 L1 节点出发，沿 MENTIONS 边找到 L2 entity
- 再反向找到提到同一 entity 的其他 L1 episode
- 解决"用户说数据库选型，但上次用的是 PostgreSQL"的语义 gap

### 3.3 效率保证

| 优化 | 效果 | 状态 |
|---|---|---|
| 批量压缩 | 10 条 turn 调 1 次 LLM → API 调用降 90% | ✅ |
| 增量索引 | 只 embed 新内容，已有内容不复算 | ✅ |
| 向量缓存 | 召回结果缓存 60 秒，重复查询零成本 | Phase 2 |
| Session 锁 | 避免重复压缩，CPU 和 LLM 调用双节省 | ✅ |

**注入格式**:

## 6. 个人 vs 团队协作记忆

### 6.1 个人模式（v0.3）✅

- `home=~/.graphmem`，单机存储
- scope = 项目名（如 `proj-backend`）
- 所有记忆只属于一个用户

### 6.2 团队模式（v0.5+，设计预留）

```yaml
# config.yaml 团队模式
mode: "C"
llm:
  driver: "deepseek"
  api_key: "..."
server:
  endpoint: "https://graphmem.internal.company.com"
  team_id: "platform-team"
  user_id: "zhangtianwei"
```

**团队记忆特性**:
- **共享 L1/L2**: 项目级 episode 和 entity 全员可见
- **个人 L0**: 原始对话只有本人可见
- **冲突检测**: 两个人对同一技术给出矛盾建议时自动标记
- **权限**: 读公开 / 写自己的 / 管理员可删

## 8. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Claude Code MCP 支持变动 | 中 | 插件失效 | MCP 是标准化协议，fallback 到 CLI |
| Embedding 模型体积大 | 低 | 启动慢 | `bge-small-zh` 只有 100MB，可接受 |
| Kuzu 并发限制 | 中 | Daemon 高并发时卡死 | Kuzu 是嵌入式，daemon 内单线程访问，HTTP 层并发 |
| LLM API 成本 | 中 | 长期使用费用高 | 批量压缩降 90% 调用量；可选本地小模型 fallback |
| 数据隐私 | 低 | 敏感代码/决策泄露 | 本地存储默认，可选自托管 server |
